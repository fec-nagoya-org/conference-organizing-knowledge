Wi-Fi

会場の Wi-Fi が弱かったり、キャパが足りない場合は、会場の許可を得たうえで回線を増設したり、 AP を設置する必要が出る場合もある。これは専門的な知識が必要となり、施工や管理に必要なスタッフもそれなりの人数が必要となる。

この場合、運営の中に Wi-Fi チーム、いわゆる NOC (Network Operation Center) チームを作り、組織的に動くことになるだろう。本章では実際のネットワーク設計や設定については割愛し、Wi-Fi 担当者以外も把握しておきたい「何が必要でどういう動きが発生するのか」にフォーカスして解説する。

# カンファレンス Wi-Fi NOC は何をやっているのか

既設設備のボトルネックが Wi-Fi 設備しかない、それより上流は心配無用という非常に稀有な例を除いては、インターネットまでの足を自分たちで確保し、そこから末端の Wi-Fi アクセスポイントまで全て自分達で面倒を見るのが一般的だろう。つまり、増設というより、新規にネットワーク設備一式を一時敷設するイメージだ。基本的に会場既設の設備はアンコントローラブルな存在で、なるべくその影響を受けないようにする必要がある。

下見の際一見問題がないように見えても、実際に多数の来場者に利用してもらった結果何かしらの問題が出るというのはよくある話だ。無駄に博打をする必要はないため、基本的には全て用意することになる。

したがって、回線・機器の調達、設計と設定、敷設・運用、撤去ということになる。

# Wi-Fi を自前で用意するために必要なもの

以下を最低限用意できないと難しいだろう。

まず、十分な帯域がある回線・ISP。といっても、1200 人でも 300Mbps も出れば十分…。ただ、運営からコントロールできない会場既設機材の影響を極力受けないよう、NTTフレッツ光などを別途調達・工事させてもらうのがポピュラー。話が分かる会場だとインターネット直結の回線をそのまま貸してくれる場合もあるが、そんなに多くはない。間にコントロール不可能な設備を通る場合は VPN といったトンネルを構築してインターネットに出るように設計するのが無難。

次にネットワーク機器・サーバ類。

DNS リゾルバ/DHCP サーバや監視サーバ、ルータ・スイッチ・Wi-Fi アクセスポイント(+コントローラ)。サーバ類は Amazon EC2 とかの IaaS に置けるし、ルータも含め、なんならミニPCでも良い。一番調達が大変なのは、数が必要な Wi-Fi アクセスポイント類になるだろう。

最後に、運用や構築が出来る人員だ。

サーバや機器のセットアップが出来る人員は、(多いに越したことはないが)実は知識さえあれば最低限でも何とかなる。一方で、会場の広さにもよるが、ケーブルを引いたり Wi-Fi アクセスポイントを各所に設置したりといった人員は、一定数必要になるだろう。

ケーブルの取り回しという点で若干テクニックは必要だが、これは現地で教えるのも容易いスキルなので、やる気があれば機器の運用をするメンバーよりかは確保がしやすいだろう。首都圏での開催であれば、ケーブル敷設や機器設置については LAN 工事業者を呼んでお願いする (敷設のみで25〜50万円くらい) という手も考えられる。

冗長化を考えず安価にすませる例:

- ルータ: NEC UNIVERGE IX2215 やミニ PC + Linux
- スイッチ: Cisco Catalyst WS-C3560CX-12PC-S
- Wi-Fiアクセスポイント: Cisco や Aruba, Juniper Mist 製品 (集中管理できるもの)
- 回線/ISP: NTTフレッツ光 + [Infal](https://www.telhi.co.jp/service/infal/hikari.html) 

(Disclosure: Infal は本章筆者の知人が経営しているサービスである。あくまで一例ではあるが、この手のイベント利用といった特殊な事例に対しフレキシブル・カジュアルに対応してくれるだろう。)

# 回線・機材・ケーブル類の調達

## 回線の調達

日本は恵まれているため、基本的には NTT フレッツ光をだいたいどこにでも引くことができる。話が通じる会場であれば、運がよければ既設の回線をONUから貸してくれることもある。この場合はキャリアに直で繋がり、会場設備の影響を受けないため便利に利用できるだろう。

フレッツ光であれば ISP を別途調達する必要がある。

容易に調達可能な PPPoE 方式は速度が遅くなりがちな一方、IPoE 方式の ISP (=VNE) は契約を回線に直接紐付ける方式を取る関係で、一時敷設した回線に即日開通が難しかったりする。この際、ポピュラーな選択肢としては同じ NTT 地域会社の管内 (東日本・西日本)にあらかじめインターネット接続とフレッツ光を用意した拠点を用意し、そこまでは ISP が不要なフレッツ光の網内折り返しを利用してしまう方法だ。

当日に設定しなければいけない機材も減らせるため、便利な選択肢と言える。

なお、実際の IPv4 インターネットへの出口も、フレッツ光の IPoE 方式にするのであれば、個人向けプランではセッション数が到底足りないため、固定 IP 付きプランが必要な点は注意したい。

費用については、フレッツ光を例に挙げるなら、1回線あたり工事費・利用費あわせて 3～4 万程度、そこに ISP の費用が乗ってくる。

## 機材の調達

先に書いたように、Wi-Fi アクセスポイントやスイッチ以外はなんとかしやすい状況にある。

たとえば、ルータについても中古のそこそこのスペック(NAPT セッション数が多くできるもの) を選べば事足りるだろうし、なんならミニ PC を Linux ルータとして使っても、問題なく捌けるだろう。

問題は数が必要になる Wi-Fi アクセスポイントやそれを繋げるスイッチだ。

まとまった数・同じ機種で調達するのはなかなか難しい。基本的にはどこかの会社から余っている機材を借りる、専門のレンタル業者から借りる、自前で買ってしまう、という選択肢がある (本章の筆者はレンタル業者は利用したことがないが、代表的な会社はキッセイコムテックなどが挙げられる)。

基本的に機材を借りる場合、業者から頼むにも、人伝の会社に融通してもらうにも、希望する物が 100% 同じ機種で揃えるのが難しいことがある。人伝だとよりランダム性・見通しの悪さがリスクになるだろう。

複数年で運用する、やる気がある場合はいっそのこと新品を購入してしまうという手もある。ハイエンドなアクセスポイントやコンパクトなスイッチはだいたい 1 つ 10～15 万円程度だ。500～650 万円くらいあれば、十分な機材を得ることができるだろう。

## ケーブルの調達

部屋に設置したスイッチからアクセスポイントまでのケーブルは、大量 (計1-2kmレベル) に必要になる。事前に下見をした際に図面も合わせて設置場所を検討、ケーブル長を算出・測定した上で調達する。

部屋間、すなわちルータ～スイッチ間のケーブルについても必要になる。運がよければ会場の LAN コンセント・パッチパネルをそのまま借用できる場合があるが、会場既設のスイッチを通過するのが不可避だったり、そもそもなかったり、そんなおいしい話はあまりない。その場合は自分たちで部屋間も引くことになる。

基本的には UTP Cat5e/Cat6 で十分だが、部屋間は距離があると光ファイバーが必要になるかもしれない。光ファイバー含め、雑に買うなら筆者は [FS.com](http://fs.com) をオススメする。

UTP ケーブルについては同じく [FS.com](http://fs.com) から買うか、商用向けのもの、たとえば日本製線・富士精線の製品を代理店から購入するのがよいだろう。

既製品にない長さも事前に作って検査して納品してくれるのでオススメだ。逆に、量販店にあるような民生品は、数を買うのが難しい・値段の割に品質が低いのでおすすめしない。

また、ドラムでまとめて買って自分たちで終端する (いわゆる「かしめる」) という手ももちろんあるが、これも品質を一定維持するのが難しくおすすめしない。現地でケーブル製作をする利点は、その場で長さを調整できることくらいだろう。

筆者が担当する RubyKaigi では例年 30～50 万円分程度を購入してケーブルは廃棄している。丁寧な撤去や保管場所の確保ができるのであれば使い回すのも不可能ではないだろう。

最後に、調達全般を通して、オンプレよろしくリードタイムがあることは十分留意する必要がある。特に工事が絡む内容は会期 5ヶ月前、それ以外も会期 3ヶ月前など、早めに発注をしておきたい。

# 敷設・撤収作業

会場規模や、どこまで自分たちで設置しなければいけないか・どれくらい人員がいるかによるが、複数トラックに渡るようなカンファレンスであれば 1 日がかりの仕事になると思って良い。

それを前提に設営日・時間を十分確保しておく必要がある。首都圏の会場では、予算都合でまる 1 日確保が難しくても、たとえば一部の部屋だけ朝から…といった工夫でなんとか時間を確保したいところ。

基本的には、朝に機材を受け取り、機材のセットアップやケーブルの設置を分担して開始する。気付けば夜になっている…という事が多い。

撤去・撤収については終演から搬出時間の間に終わらせる必要がある関係で、敷設よりも時間的にシビアになりがちだが、機器さえ回収してしまえばケーブルの回収は NOC チーム外からも協力してもらうなどで迅速に終わらせることができるだろう。

ここは廃棄前提であれば雑に回収しても OK という前提だ。また、多数のゴミを出すことになるため、事前にファシリティを見ている関係者や会場に内容や量を伝えて、ゴミ回収の調整は忘れないようにしたい。

# 搬入と搬出

基本的には、どこかで機材をまとめて準備してから、会場に送ることになる。

一般的な宅配便では午前指定といったゆるい指定しかできないため、赤帽や JITBOX チャーター便といった発送方法を使う。

特に精密機器が多数あるため、宅配便より安心? できるという点もポイントだろう。その観点では、なるべく物理で持たないといけない・移動させないといけない物は最小限にしておきたい (サーバーをクラウドに置くなど)。

RubyKaigi では、倉庫から荷物を誰かのオフィスに取り出し、一時保管・整備の上会場に JITBOX で送り直している。

搬出については同様の方法で倉庫に直接戻してもいいし、機材の調達方法によっては一度オフィス等に戻して、借りた機材を宅配便などで各々に返すといった事が必要になる。

撤収作業は先に書いたように時間的にシビア…というか夜間になるため、宅配便の集荷が間に合わない。

首都圏の場合は赤帽、地方の場合は必然的に JITBOX チャーター便になるが、赤帽はともかく JITBOX の集荷にも大抵間に合わない。

そのため、場合によっては翌営業日まで会場内の会議室を借りて荷物を残置して、翌営業日に発送…といった手間が必要になる。事前に会場や業者と確認しておきたいポイントだ。

## 倉庫

Wi-Fi に限らないが、備品や機材を保管する必要が生じてくる。

私物の持ち寄りなら困らないが、もしイベントで買った物を保管するとなると、法人向けの倉庫サービスや、メンバー個人がたまたま戸建とかを持っていて保管してる…とか様々な例を聞く。

法人向けの倉庫サービスだと普通断られる精密機器類も、契約を調整して利用者責任の元預けられたりするので問合せてみるのがおすすめ。

# NOCチームを組みたい・実際の事例

難しいことを書くと、ここは正直いい伝手がないと上手くいかないだろう。

本章筆者が率いる RubyKaigi NOC では [RubyKaigi](https://rubykaigi.org), [Kaigi on Rails](https://kaigionrails.org) での Wi-Fi 構築を担当している。

基本的には少数精鋭で、かつ L3 以上のネットワーク設計が必要なところは、スキルがある人間が毎年何かしらの取り組みをするといったモチベーションを持って、基本似たメンバーで継続している。

これは手前味噌ながらも、数少ない事例だろう (参考:  [RubyKaigi NOC 近況 2025](https://speakerdeck.com/sorah/rubykaigi2025wifi))。

基本的には、カンファレンス本題のコミュニティにも関心がある人が集められると良い。たまに Wi-Fi 班の詰め所みたいなところに、ずっと多人数いるカンファレンスとかを見るとちょっと悲しくなる (特に学生)。

昔は [CONBU](https://conbu.net/), 近年では [BAKUCHIKU](https://note.com/sasakipochi/n/n0786b6c9b4ee) といった Wi-Fi 提供を専門とするチームや、そこから派生した NOC は数少なくない。

だいたい似たようなコミュニティのメンバーで回っている気はするが、それが示す通りコミュニティ業でカンファレンス Wi-Fi が出来る人は、そもそもそんなに多くないのだ。

近年では(実際に利用できるネットワークを構築できる!というような名目で)学生を募って、スポンサーから機材を調達して 0 から毎回構築して提供するような NOC を多数見る。

0 からその回で集められたランダムな機材で構築するというのは、一発勝負になって安定度に欠けるものだ。

構築するのが若手メンバーというのであれば尚更になる。イベントの印象、参画してくれたメンバーのメンタル、Wi-Fi スポンサーのレピュテーション…など「不安定だった」という結果に終わるのは、当然ながらあまり良いことではない。

最低1人でもいいので、スキルとノウハウがあるメンバーを確保して、品質を最終的になんとかする動きができるチームにはしておきたいところ。

また、セキュリティやプライバシーといった概念の理解は最低限求めたい。

[PyCon APAC 2023 で DNS クエリ統計をそのまま公開した事例](https://pyconjp.blogspot.com/2023/11/pyconapac2023-statement.html)は記憶に新しいが、誤った取り組みや行動はイベントそのもののレピュテーションにかかわってくる。この点でも取り組みや品質についてレビューできる体制はかならず必要と言えよう。

# 最後に: いっその事「提供しない」という選択肢

ここまで読んだ多数の感想は「大変そう」だと予想する。実際そう思う。

近年では、セルラーも4G/5Gときて快適になってきているし、USB テザリングであれば無線区間の心配はあまりしなくていいため、参加者には Wi-Fi なしで乗り切ってもらうというのも、十分とれる選択肢だ。

海外の中規模カンファレンスくらいでは、Wi-Fi 全然頑張ってなくて会場設備で使い物にならない、というのはすごくよくある話だし。なんとかなる。

この文章の内容を頭の隅に置いて「自前で構築する場合に必要なこと」を何となく把握しておきつつ、いつかコミュニティ内でモチベーションがある人がいたら、そうでなくても良い人がいればお願いしてみてほしい。

多数の人間が利用するネットワークを構築する機会は現代では貴重で、この文書を踏まえてそのような場を提供するぞ! としてくれる主催者はありがたいし、応援します。

# FAQ

## Q. そもそも会場の Wi-Fi が弱いって何?

ざっくり、大抵の既設設備ではボトルネックはだいたい以下に収まる。

(だれも 1000 人きて全員がパソコンカタカタすることは想定していないのだ…。)

- 設置されているアクセスポイントの数

- \= 最大同時接続数になる。
- ハイエンド機で70～80接続/APくらいを目安にしたい

- アクセスポイントごとのチャンネルや出力レベルの設定

- 同じ設備同士でチャンネル干渉してないか
- むやみに出力レベルが高くないか
- 無駄にボンディングしていないか

- 回線の実効帯域

- 1,200 人でも 300Mbps あれば十分だけど、そもそも裏がフレッツの PPPoE で遅すぎるとかがあるかも

- NAPTセッション数、タイマー設定

- 特に最近は QUIC により UDP の NAPT セッション数を多数消費してしまうという問題があるので、タイマー設定も特に重要

- サブネット、DHCPプールサイズ

- サイズがたりてないと繋げてもIPアドレスが得られなくて外に出れなくなっちゃう

会場に「1,200 人きて全員つなぎますよ」と伝えて「絶対大丈夫です!」と言われても過信してはいけない。

事前に伝えたのにもかかわらずサブネットサイズが /24 (254 アドレス) で DHCP プールが即座に枯渇した、みたいな例があったりもするし、基本的には信用できないものと思おう。

## Q. 法人契約しかできない内容が含まれてない?

その通り。

一部特殊な回線の契約、機材の保管や商用向けネットワーク機器の購入、また搬入出に利用する物流キャリア、様々な点で法人格が前提、もしくはあった方が捗ることが多い。法人格については スポンサー の章を参照されたい。

## Q. Wi-Fi スポンサー・ネーミングライツってどう思う?

基本的には覚悟を伴う。

単にお金を出すだけのスポンサーであればオススメはしない。

機材か人員、どちらかはお願いしたいところだ。というのも、Wi-Fi スポンサーとして名前が出てしまうと、障害時・うまくいかなかった際にその企業のレピュテーションに関わってきてしまうためだ。

機材の提供に関しても、毎年バラバラにランダムに提供されて0から設定する…というのはそこそこ工数が掛かるし、安定性や博打度にかかってくる。

この点では、毎年開催してかつスポンサーも調達できているのであれば、機材の購入やレンタルを検討したいところだ。

## Q. いわゆるMICE向けのネットワーク業者に頼むのはどう思う?

日本では、テックカンファレンスで期待される品質の、イベントネットワークを作れる専門業者いるんだろうか?

どのみち MICE 向け・イベント向けの業者はかなり高くつく (数百万円クラス) 認識なので、それであれば「提供しない」選択肢でいいじゃん、となる気がします。

コミュニティでやったほうが面白いだろうという、筆者のバイアスはもちろんあるので、一つの意見として。
